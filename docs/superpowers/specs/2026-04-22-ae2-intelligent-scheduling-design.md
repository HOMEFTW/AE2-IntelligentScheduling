# AE2-IntelligentScheduling 设计文档

## 1. 概述

`AE2-IntelligentScheduling` 是一个面向 GTNH / Minecraft 1.7.10 / AE2 的新模组，目标是在 **不影响 AE2 原有合成按钮与原有功能** 的前提下，在 AE2 原合成 UI 上新增一个 `智能合成` 按钮。

玩家点击 `智能合成` 后，系统不再直接走 AE2 默认的一次性下单路径，而是：

- 读取当前目标物品与数量
- 分析 AE2 合成树
- 先扣除 AE 网络当前已有库存
- 对整棵树中达到阈值的中间产物与最终产物执行拆分
- 生成按依赖分层的合成队列
- 按层程序性地下单到 AE2 的合成 CPU
- 在下层缺口完成并可被 AE 网络消费后，自动继续下一级

本模组本质上是一个 **AE2 智能合成调度器**：AE2 仍负责单个 job 的实际执行，本模组负责大体量任务的分析、拆分、排队、依赖控制与自动推进。

## 2. 目标与非目标

### 2.1 目标

- 保留 AE2 原 `合成` 按钮和原有下单行为
- 在 AE2 原合成 UI 中新增 `智能合成` 按钮
- 递归分析整棵 AE2 合成树
- 对中间产物与最终产物都支持智能拆分
- 拆分前先扣除 AE 网络现有库存，仅为缺口建队列
- 按依赖顺序分层调度，先下底层，再下上层，最后下最终产物
- 自动使用当前空闲的可用 CPU，不要求玩家预先指定
- 让智能订单在关闭 GUI 后仍继续运行
- 为失败、暂停、取消、重试提供明确状态反馈

### 2.2 非目标

- 不修改 AE2 原有 `合成` 按钮语义
- 第一版不做跨服务器重启的运行中任务无损恢复
- 第一版不做复杂的整棵树图形化可视化
- 第一版不强行接管普通 AE2 合成流程

## 3. 用户确认的关键需求

### 3.1 交互入口

- 功能载体：AE2 网络部件 / 终端
- 保留 AE2 原合成 UI 与原 `合成` 按钮
- 在原合成 UI 上新增 `智能合成` 按钮
- 点击 `智能合成` 时加载本模组逻辑

### 3.2 数量单位与拆分阈值

- `1g` = `1,000,000,000` 个物品
- `2.1g` = `2,147,483,647` 个物品，即 `Integer.MAX_VALUE`

拆分规则：

- 缺口 `< 1g`：使用 `1` 个 CPU 执行
- 缺口 `>= 1g` 且 `< 4g`：使用 `2` 个 CPU 执行
- 缺口 `>= 4g` 且 `< 16g`：使用 `4` 个 CPU 执行
- 缺口 `>= 16g` 且 `< 64g`：使用 `8` 个 CPU 执行
- 缺口 `>= 64g`：使用 `16` 个 CPU 执行

实际分配规则：

- `实际分配 CPU 数 = min(规则建议值, 当前空闲 CPU 数, 配置允许的单节点最大 CPU 数)`
- `2.1g` 仍作为数据边界与溢出风险参考值保留，但不再作为 CPU 拆分分档阈值

### 3.3 拆分范围

- 对整棵树中所有达到阈值的中间产物和最终产物都递归拆分

### 3.4 库存语义

- 智能合成分析时，先扣除 AE 网络当前已有库存
- 只对缺口部分生成节点、队列与下单任务

### 3.5 调度语义

- 必须按依赖顺序分层下单
- 先下最底层中间产物
- 等下层达到可用数量后，再自动下上一级
- 最后再下最终产物

### 3.6 CPU 选择

- 自动挑选当前空闲的可用 CPU
- 不要求固定某几颗 CPU

## 4. 总体架构

推荐实现路径为：

- `Mixin + 智能终端按钮 + 独立调度器内核`

内部拆分为三层：

### 4.1 UI 接入层

职责：

- 在 AE2 原合成 UI 上新增 `智能合成` 按钮
- 读取当前玩家准备合成的目标对象与数量
- 打开本模组的智能订单状态面板

约束：

- 原 `合成` 按钮行为保持不变
- 仅点击 `智能合成` 时进入新逻辑

### 4.2 规划分析层

职责：

- 调用 AE2 合成规划接口获取原始合成树或可遍历计划
- 遍历整棵树并抽取中间产物与最终产物节点
- 查询 AE 网络库存并扣减已有量
- 识别阈值命中节点并生成拆分策略
- 将树结构拍平成按依赖分层的任务队列

### 4.3 调度执行层

职责：

- 按层调度 `SmartCraftTask`
- 监控 CPU 空闲情况并提交任务
- 跟踪 AE2 任务句柄与库存变化
- 在当前层完成后自动推进下一层
- 对失败 / 取消 / 暂停 / 重试进行统一管理

## 5. 核心数据模型

### 5.1 SmartCraftNode

表示合成树中的一个“缺口节点”。

建议字段：

- `requestKey`：产物唯一标识，支持 item / fluid
- `displayName`：显示名称
- `requestedAmount`：理论总需求
- `availableAmount`：AE 当前可用库存
- `missingAmount`：扣库存后的实际缺口
- `perCraftOutput`：单次配方产出量
- `craftable`：AE2 是否可合成
- `splitRequired`：是否达到拆分阈值
- `suggestedCpuCount`：建议 CPU 数，取值 `1 / 2 / 3`
- `depth`：在依赖树中的层级深度
- `children`：子节点列表

### 5.2 SmartCraftTask

表示实际提交给 AE2 的一个任务片段。

建议字段：

- `taskId`
- `orderId`
- `layerIndex`
- `requestKey`
- `targetAmount`
- `expectedOutputAmount`
- `assignedCpuCount`
- `status`
- `aeCraftingLinkRef`
- `failureReason`
- `createdAt`
- `startedAt`
- `finishedAt`

### 5.3 SmartCraftLayer

表示某一依赖层上的全部任务。

建议字段：

- `layerIndex`
- `tasks`
- `status`
- `blockingReason`

### 5.4 SmartCraftOrder

表示一次完整的智能合成订单。

建议字段：

- `orderId`
- `playerUuid`
- `targetRequestKey`
- `targetAmount`
- `status`
- `layers`
- `currentLayerIndex`
- `summary`
- `createdAt`
- `updatedAt`

## 6. 合成树分析与分层规则

### 6.1 分析流程

1. 玩家点击 `智能合成`
2. 读取当前 AE2 UI 中选中的目标产物与数量
3. 调用 AE2 规划接口生成原始合成计划
4. 递归遍历整棵树
5. 对每个节点查询 AE 网络现存库存
6. 用 `requestedAmount - availableAmount` 得到 `missingAmount`
7. `missingAmount <= 0` 的节点不进入待执行队列
8. `missingAmount > 0` 的节点进入 `SmartCraftNode`
9. 根据阈值计算是否拆分与建议 CPU 数
10. 按依赖深度拍平成 `Layer 0..N`

### 6.2 分层规则

- `Layer 0`：最底层、没有子依赖的缺口任务
- `Layer 1`：依赖 `Layer 0` 的中间产物
- ...
- `Layer N`：最终产物任务

父节点是否可执行，不取决于“子节点理论上已提交”，而取决于“子节点对应任务片段已经完成并确认可被 AE 网络消费”。

### 6.3 拆分规则

拆分始终发生在 **扣除库存之后**。

- `missingAmount < 1,000,000,000`：生成 `1` 个 task
- `1,000,000,000 <= missingAmount < 4,000,000,000`：生成 `2` 个 task
- `4,000,000,000 <= missingAmount < 16,000,000,000`：生成 `4` 个 task
- `16,000,000,000 <= missingAmount < 64,000,000,000`：生成 `8` 个 task
- `missingAmount >= 64,000,000,000`：生成 `16` 个 task

拆分算法要求：

- task 数量之和必须严格等于 `missingAmount`
- 若无法整除，则将余数依次分配给前面的 task
- 不允许因整除误差导致总量丢失或溢出
- 实际提交时还要再次裁剪到“当前空闲 CPU 数”和“配置允许的单节点最大 CPU 数”

## 7. 执行状态机

### 7.1 SmartCraftTask 状态

- `PENDING`：尚未开始
- `WAITING_CPU`：当前层可执行，但没有足够空闲 CPU
- `SUBMITTING`：正在向 AE2 提交 job
- `RUNNING`：AE2 已接受并执行
- `VERIFYING_OUTPUT`：校验产物是否已真正进入 AE 网络
- `DONE`：任务完成
- `FAILED`：任务失败

### 7.2 SmartCraftOrder 状态

- `ANALYZING`：分析合成树
- `QUEUED`：已生成队列，等待执行
- `RUNNING_LAYER_n`：正在执行第 `n` 层
- `PAUSED`：暂停
- `COMPLETED`：全部完成
- `FAILED`：整单失败
- `CANCELLED`：整单取消

### 7.3 状态迁移规则

#### Task

- `PENDING -> WAITING_CPU`
- `WAITING_CPU -> SUBMITTING`
- `SUBMITTING -> RUNNING`
- `RUNNING -> VERIFYING_OUTPUT`
- `VERIFYING_OUTPUT -> DONE`
- 任意执行态 -> `FAILED`

#### Order

- `ANALYZING -> QUEUED`
- `QUEUED -> RUNNING_LAYER_0`
- `RUNNING_LAYER_n -> RUNNING_LAYER_(n+1)`
- 最后一层完成 -> `COMPLETED`
- 某个关键任务失败 -> `PAUSED` 或 `FAILED`
- 玩家取消 -> `CANCELLED`

## 8. 调度规则

### 8.1 层推进

- 仅当当前层全部 task 达到 `DONE` 时，才允许推进到下一层
- 不允许上层任务在下层未完成时提前提交

### 8.2 CPU 选择

- 自动挑选当前空闲的可用 CPU
- 不要求玩家手动指定 CPU
- 单个节点的目标 CPU 数由分档规则得出，范围为 `1 / 2 / 4 / 8 / 16`
- 若当前没有足够空闲 CPU，则 task 进入 `WAITING_CPU`
- CPU 重新可用后，调度器自动继续尝试提交

### 8.3 完成判定

一个 task 的完成不只依赖 AE2 内部 job 完成信号，还需要二次确认：

- 记录该 task 预期应补足的产物量
- 周期性重新查询 AE 网络库存
- 只有当对应产物的可用量净增长满足该 task 的目标时，task 才进入 `DONE`

该设计优先保证“上一级下单时，下一级依赖产物已经确实可被消费”。

### 8.4 失败处理

默认策略：

- 某个 task 失败时，暂停整单
- 在 UI 中展示失败节点、失败层级、失败原因
- 玩家可选择：
  - `重试失败任务`
  - `取消整单`

第一版不自动跳过失败节点继续推进上层。

## 9. UI 设计

### 9.1 AE2 原 UI 改动

在 AE2 原合成 UI 上新增 `智能合成` 按钮。

约束：

- 原 `合成` 按钮保留原样
- 原有 UI 功能不被替换
- `智能合成` 与原按钮逻辑完全分离

### 9.2 智能订单面板

点击 `智能合成` 后，打开本模组的状态面板或弹窗。

建议显示内容：

- 目标产物
- 目标数量
- 订单状态
- 当前执行层号
- 当前层任务列表
- 每个任务的目标量、拆分片数、状态
- 占用中的 CPU 数量
- 等待 CPU 的任务数量
- 失败任务与失败原因

建议操作按钮：

- `取消整单`
- `重试失败任务`

### 9.3 第一版 UI 范围

第一版不做复杂树图，仅做：

- 按层显示
- 当前层重点展示
- 上下层摘要可展开查看

原因：

- 超大合成树节点过多，整树图在 GTNH 中可读性差
- 第一版优先保证状态可追踪与操作可恢复

## 10. 持久化与恢复

### 10.1 运行期行为

- 智能订单是服务端状态
- 玩家关闭 GUI 后，订单继续执行
- 玩家重新打开终端时，可重新查看当前订单状态

### 10.2 服务器重启策略

第一版策略：

- 不做运行中智能订单的无损恢复
- 服务器重启后，将上次运行中的订单标记为“中断”
- 玩家可以看到中断订单摘要
- 若玩家要继续，系统重新分析当前 AE 网络状态并生成新队列

不在第一版实现范围内的内容：

- 持久化整棵运行中任务树
- 重启后重新绑定 AE2 原有运行中 job 句柄
- 在重启后继续沿用旧的 CPU 占用关系

- `integration.ae2.ui`
  - AE2 GUI 注入
  - 智能合成按钮入口

  - AE2 合成规划读取
  - 原始树适配层

  - `SmartCraftNode`

  - `SmartCraftLayer`
  - `SmartCraftOrder`

  - 缺口分析器
  - 库存扣减逻辑

  - 智能订单状态面板

## 12. 风险与边界

### 12.1 AE2 API 可观测性风险

AE2 1.7.10 的规划树与运行中 job 可观测性可能不完全满足需求，可能需要：

- 对现有 API 做适配封装
- 使用 Mixin 或反射补足部分内部状态读取

### 12.2 超大数量与整数边界

必须特别注意：

- `long` 与 `int` 的边界转换
- 拆分时的守恒
- AE2 / UI / 网络同步字段是否存在 `int` 限制

### 12.3 外部并发消耗

在智能订单执行过程中，AE 网络中的中间产物可能被其他系统消耗。第一版通过“完成前二次库存校验”来缓解该问题，但仍需在实现与测试阶段重点验证。

### 12.4 CPU 资源竞争

如果其他玩家或系统同时占用 CPU，任务会停留在 `WAITING_CPU`。这是预期行为，不应视为逻辑错误，但 UI 必须清楚展示阻塞原因。

## 13. 测试策略

### 13.1 单元测试

- 库存扣减逻辑
- 合成树转分层队列
- `1g / 4g / 16g / 64g` 分档拆分算法
- task 与 order 状态机迁移
- 余数分配与总量守恒

### 13.2 集成测试

- AE2 规划结果适配
- job 提交流程
- CPU 选择逻辑
- job 完成 / 失败回调
- 基于库存增长的完成确认

### 13.3 游戏内烟测

- 原 `合成` 按钮功能未被影响
- `智能合成` 按钮可见并可点击
- 大量中间产物能按层拆分与推进
- CPU 不足时能等待并在空闲后继续
- 失败任务能暂停整单并允许重试或取消

## 14. 验收标准

实现完成后，应满足以下验收条件：

- AE2 原有合成按钮与原流程完全可用
- `智能合成` 能从原合成 UI 正常进入
- 能递归分析整棵树并扣减库存
- 对中间产物和最终产物都能按阈值拆分
- 调度严格按层推进
- CPU 不足时任务进入等待，不丢失
- 中间产物完成后能自动推进上一级
- 失败时整单暂停并暴露原因
- 玩家可取消整单或重试失败任务

## 15. 实施范围建议

建议将后续实现拆为多个波次：

- Wave A：项目脚手架、AE2 依赖接入、基础按钮注入、订单模型
- Wave B：合成树分析、库存扣减、拆分器、分层队列
- Wave C：调度器、CPU 选择、状态机、任务跟踪
- Wave D：状态 UI、失败恢复、烟测与稳定性修复

本设计文档对应的是 **完整目标设计**，不代表第一波实现必须一次完成全部功能。
